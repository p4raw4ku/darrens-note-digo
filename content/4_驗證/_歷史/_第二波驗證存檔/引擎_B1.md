## 二、三處覆寫的實作說明（律師卷宗硬約束 1）

### (a) 過載計時：絕對回合數、離場不凍結（BL-1）

dir1/dir2 原版（`setups.Dir2Engine.hook_end_turn`）用 `st.config.extra
['_overload_pending_%d'%side]` 做二段式切換，隱含「現在 `st.units[side]` 是誰」
的假設——換角把單位換下場後，這個切換機制對「誰觸發了 hook_end_turn」失去追蹤，
效果等於離場即凍結（律師卷宗 BL-1 讀法B 的成因）。

B1 完全不使用這套機制。`B1Engine.hook_end_turn` 對過載無事可做（故意不呼叫
`super().hook_end_turn()`）；改在 `_b1_resolve_attack()`（薄殼包 `core.resolve_attack`）
內，每次傷害結算前後直接讀寫 `defender.overload_until_round`（絕對回合數，`None`＝
未過載）：

```python
if defender.overload_until_round is not None:
    if cur_round > defender.overload_until_round:
        defender.field_disabled = False
        defender.overload_until_round = None
    else:
        defender.field_disabled = True
...
if defender.field_disabled and defender.overload_until_round is None:
    defender.overload_until_round = cur_round + 1
```

到期回合數是 Unit 物件上的欄位，換角＝物件被搬動，離場期間 `st.round_no` 照樣推進、
下次結算時一比較即知是否已過期——不需要在換角當下做任何「凍結」或「解凍」動作，
自然達成「離場不凍結」。手工測試 `test_4_preserve_overload_absolute_timer` 驗證：
換角瞬間到期回合數原封不動、離場期間回合數推進 3 輪後到期即解除。

### (b) 主格深拷貝保留（BL-5）vs B 變體全重置（BL-6）

`B1Engine.hook_apply_change_navi`：

```python
if self.variant == 'A':
    pass   # 不 touch 任何防禦欄位——這正是「保留」的字面意義
else:
    bench_unit.shield_cur = bench_unit.shield_max
    bench_unit.field_disabled = False
    bench_unit.overload_until_round = None
    bench_unit.pierce_pending = False
```

A 讀法刻意「什麼都不做」：`bench_unit`（即將上場的後備）本來就是同一個 Python
物件、離場期間沒被任何程式碰過，`shield_cur`／`field_disabled`／
`overload_until_round`／`pierce_pending` 全部原樣帶回——「深拷貝保留整個防禦狀態
物件跟人走」在實作上等於「換角不重置」。B 變體照抄 dir1 v0 語義，但額外補上
`overload_until_round` 與 `pierce_pending` 的清除（dir1 v0 原版沒有這兩個欄位，
律師卷宗 BL-6 要求 B 變體是「一整包」重置，故補齊，否則 A/B 對照會不乾淨）。

### (c) 穿透旗標跟人（BL-3/BL-5）

共用核心的護盾狀態機（`shield_cur==0` 即代表「下一擊全額穿透」）本來就是逐擊判斷
`defender.shield_cur`，這個欄位天然跟著 Unit 物件走——換角＝物件搬動，不需要另外
攔截。B1 新增 `pierce_pending` 作為這個中間態的**顯式**旗標（供測試與後續指標統計
讀取「穿透待發」狀態，不是改變傷害結算本身），在 `_b1_resolve_attack` 結算後同步：

```python
defender.pierce_pending = (defender.shield_cur == 0 and defender.shield_max > 0)
```

主格 A 讀法下這個旗標跟著 `_B1Unit` 物件離場再上場；B 變體則在 `hook_apply_change_navi`
被清空。手工測試 `test_4_preserve_pierce_flag` 驗證：離場時盾0＋旗標亮 → 換角後
第一擊（過力場後）全額穿透、之後護盾回滿。

## 三、其餘裁量條款落地

- **陷阱敵半場＋換角出現格不觸發／不拾取**（BL-4／BT-2）：`hook_apply_change_navi`
  刻意不呼叫 `hook_trap_check`／`hook_resource_pickup`（與共用 `apply_action` 的
  「移動」分支對照，那裡兩者都會呼叫）。手工測試在中央格布陷阱＋資源球，驗證強制
  換入者出現該格兩者皆不觸發。
- **安全網只扣上場者＋方級豁免＋遞增制**（BL-2）：`apply_safety_net` 整個覆寫（共用
  核心版是寫死 `-10`），改成 `amount = base + step × elapsed`，只對 `st.units[side]`
  扣血、豁免判準沿用 `st.damage_this_turn[side] > 0`（跟方不跟單名，後備不受影響）。
  手工測試驗證起點/基值/步幅（15/5/4 → 5,9,13,17…）與方級豁免。
- **換角 Action 僅兩活可換**（BR-5）：`hook_change_navi_actions` 只在存活後備存在時
  產生 Action；一名倒下後自動非法（無需額外禁止規則）。強制換入（`hook_on_unit_down`）
  純被動、不經過行動經濟（`actions_this_turn` 不記錄），出現在我方中央格
  `(2,2)`/`(5,2)`。
- **費2 旗標 per-side、換角不清**（BR-4）：沿用 `Dir2OverlayEngine` 既有的
  `_hits_bonus_pending_%d % side` 機制（本來就存在 `config.extra`，未被
  `hook_apply_change_navi` touch 到）——確認不需要改動，只需要驗證換角不會意外清除
  它。手工測試構造「甲掛旗標→換乙→乙的下一次晶片能力面攻擊 hits+1 仍生效」全程。
- **主格列首先發**（BR-6）：`make_b1_setup(proto_a=('游擊','壁壘'), proto_b=(...))`，
  `protos[0]` 固定上場、其餘進 benches。
- **首回合換角合法**（BR-1）：不需額外處理，`legal_action_pairs(..., first_comp_limit='一行動')`
  在只給 1 行動時自然只讓換角出現一次（第二行動強制 pass）。
- **B6 team_hp 全隊總 HP 口徑**（BT-1）：`B1Engine.team_hp` 覆寫為上場者＋存活後備
  HP 加總。

搞反（旗標是 per-side 掛在 side0，換角後應以 side0 身份攻擊，而非誤用 side1）；
兩者修正後皆為引擎正確行為的確認，非放寬測項。

## 五、冒煙測試（greedy 鏡像 n=50）

```
variant=A n=50 耗時=5.23s 回合中位=21.0(min20/max21) 勝負局=2 平局=48 換角總次數=45 零換角局比例=0.46
variant=B n=50 耗時=5.09s 回合中位=21.0(min20/max21) 勝負局=1 平局=49 換角總次數=45 零換角局比例=0.46
```

三行摘要：
1. **不崩**：A/B 兩變體各 50 局全部正常結束（無例外、無卡死），耗時皆 ~5 秒。
2. **長度合理**：回合數集中在 20–21（安全網起點15後遞增制在數局內即分出勝負或
   雙方同歸於盡），min/max 窄幅波動，無異常拖到 `max_turns=60` 觸頂的情形。
3. **換角有發生**：50 局共 45 次主動換角、54% 的局至少換角一次，證實換角機制在
   greedy_damage 自我對戰下確實會被觸發，不是死代碼。

**觀察（非本次任務範圍、留給電池 I/II 判讀）**：greedy_damage 鏡像下平局
（同歸於盡）率偏高（A 96%／B 98%），成因初判是全隊 100 HP 對照遞增制安全網
（起點15、步幅+4）在雙方對稱決策下容易同時觸底——這是平衡現象，留給漏洞電池與
判讀欄處理，不影響本次引擎交付的正確性判準。
