# 方向一「讀心換角戰」漏洞獵殺實測報告

> 把紙上退化套路寫成真策略、在方向一模擬器上實測，判定哪些窟窿是真的。
> 策略程式：`engine/exploit_dir1.py`（不改任何共用檔）；跑分輸出
> `results/dir1/exploit_*.json`。所有數字為實跑，非推估。

## 判定標準（照任務指示）

- **成立**：對每個基線都拿下 ≥60% 勝率（不含平局）且贏法與紙上機制一致。
- **部分成立**：只對部分基線成立、或勝率落在 55%~60%。
- **不成立**：對主要基線勝率 <55%。
- **無法實作**：紙上前提在不改共用檔（core.py 等）下無法成立。

基線策略 7 種：random_legal、greedy_damage、turtle、positional（會走位）、
oneply、predictor、naive。每局漏洞策略先後手各半。**局數說明見文末「局數與環境」。**

---

## 關鍵環境事實：方向一是「安全網攤牌局」

實測所有方向一對局——**連基線互打**——都拖到約 41~46 回合、安全網
（第 25 回合起每回合雙方各扣 10HP、造傷者豁免）在 **100% 對局觸發**：

- greedy_damage vs greedy_damage（40 局）：回合中位 41、安全網觸發 40/40、擊倒 35。
- turtle vs turtle（40 局）：回合中位 42、安全網觸發 40/40、擊倒 37。

成因：每名 70HP×3＝210 總血，而防禦型之間穿透率極低（自攻對盾/力場實傷 0、
大擊對閃避 96% 蒸發、多發對力場全歸零，見相剋矩陣 `exp3_方向1_相剋矩陣.md`），
開局幾回合把手牌打完後雙方多半只能 pass 攢能量，真正的擊殺主力是安全網。

**方向一的主要獲勝條件是「誰在安全網攤牌期活得久」，不是紙上描述的開局戰術。**
這個事實決定各套路成敗：**能打破攤牌對稱性的（回復牆）成立；靠開局壓制的
（集火/全域點）反而因「主動進攻＝暴露＝被打」而輸給龜縮方——攻擊在此配置
被系統性壓制，決勝的是存活（回復/防禦）而非輸出。**

## 套路1：盾型回復牆 — 成立

**機制（隔離、已驗證）**：盾型單回合最大回復＝迴復裝置 30＋盾能力（回 25＋
護盾回滿）＝55HP；自攻對盾 25 實傷 0（相剋矩陣證）。

**對局勝率（n=1000/cheap 基線，先後手各半）**：

| 基線 | n | 勝率(不含平) | 勝/負/平 | 回合中位 | 觸頂平局 |
|---|---|---|---|---|---|
| random_legal | 1000 | **0.987** | 845/11/144 | 46 | 141 |
| greedy_damage | 1000 | **0.913** | 878/84/38 | 42 | 0 |
| turtle | 1000 | **0.984** | 981/16/3 | 44 | 0 |
| positional | 500 | **0.841** | 418/79/3 | 40 | 0 |
| oneply/predictor/naive | 小 n | （best-effort，見文末；smoke n=6 皆 ≥0.83） | | | |

**判定：成立**（對每個已完成的 cheap 基線都遠超 60%）。

**實測贏法（與紙上機制的異同）**：贏法是「盾型自我回復撐過安全網攤牌期」，
但**收局多半是擊倒、不是紙上預測的 60 回合觸頂平局**（回合中位 42~46）。機制上
盾型每回合回 25~30HP > 安全網每回合扣 10，能無限撐住上場那一名；對手三名都得
撐過安全網、而其對盾型的攻擊多半 0 實傷（無法豁免安全網、照扣 10），於是對手
被安全網逐一磨死。**只有對 random_legal 才出現大量觸頂平局（141 局，14%）**——
對手夠弱時真的拖到 60 回合觸頂。紙上「回復無限疊」的核心對，收局形式依對手
強弱在「擊倒」與「觸頂」間變動；紙上點名的症狀（觸頂率>0、安全網觸發率非極少、
讀心/naive 勝率被拉平）全部命中。

## 套路2：護盾先削後重 — 機制成立、對局內不成立

**機制（隔離、確定性、已驗證）**：
- 機關槍 5×5 削盾歸 0 ＋ 重錘 50×1 直擊＝總實傷 **50**（vs 直接重錘單發對盾 25＝**25**，穿透翻倍）。
- 三連彈 10×3 削盾 ＋ 砲能力 40×1＝總實傷 **45**（vs 單發砲 40 對盾＝**15**）。

**對局勝率**：vs greedy_damage **0.375**（175/292/33，n=500）；pilot vs turtle 0.28、
vs random 0.63（n=20）；其餘 500-game 續跑於 `exploit_shave_heavy_*.json`。

**判定：不成立（對局內）／機制成立（隔離）**。原因：先削後重要求同回合手上
**同時**有小擊削盾牌＋重擊牌，且對手上場**恰為盾型且在射程內**。對手預設隊伍
起手是斬型（閃避），只在後段輪轉才偶爾上盾型；30 張共用牌庫又稀釋了「小擊＋
重擊同手」的機率。於是這個確定性剝削在隔離結算下完全成立、翻倍穿透屬實
（存活條 3 實錘），但**在對局中觸發窗口太窄、無法轉成 ≥55% 的勝率**。

## 套路3：全域點/範圍指定集火（無視走位）— 不成立

**機制（幾何事實、已驗證）**：全域點命中任選敵方一格、不需同排或貼身
（core.cells_hit 已證），走位對它無效。狙擊槍 35×1 對盾 25 穿 10、對無防禦穿 35、
追蹤彈 15×2 對盾 25 穿 5。

**對局勝率**：pilot vs greedy 0.17~0.21、vs turtle 0.03~0.17（n=20~30）；
500-game 確認見 `exploit_global_focus_*.json`。

**判定：不成立**。全域點「無視走位」在幾何上千真萬確，但**這不構成獲勝優勢**：
在攤牌局裡進攻方每回合把僅有的幾張全域點打完後就只能 pass，而全域點對防禦型
穿透有限（對盾/力場多半 0）；主動輸出讓自己在安全網期沒有存活優勢，反被龜縮方
熬死。紙上「全域點支配走位」的症狀（移動占比趨 0）確實出現，但那是因為**整個
方向一移動占比本就趨 0**（大家都龜），不是全域點造成的差異化優勢。

## 套路4：集火滾雪球 — 不成立

**機制（隔離、已驗證）**：砲能力 40×1 秒殺盾型 70HP 需 3 發（15/40/15）、
無防禦 2 發；但對閃避 8 每發期望僅 1.44、根本秒不掉（需 ~45 發）。

**對局勝率**：pilot vs greedy 0.33、vs turtle 0.37（n=20）；500-game 確認見
`exploit_focus_snowball_*.json`。

**判定：不成立**。集火效率「極度依賴對手防禦型」這點屬實（對盾/無防禦能秒、
對閃避完全秒不掉），但在對局中：(a) 對手防禦隨換角輪轉，集火手手上未必有能穿
當前防禦的形狀；(b) 雙方傷害近乎對稱（實測一局 side0 造傷 75 vs side1 造傷 74），
勝負由安全網 coin-flip 決定。集火不需讀心即可穩定滾雪球的紙上假設**不成立**——
它連穩定贏都做不到。2對3 雪球在此配置存在（首名陣亡後強制換入不補償），但集火
方自己也被同一個安全網磨、無法把雪球滾成勝勢。

## 套路5：免費換角支配走位 — 無法實作

**紙上核心前提無法實作**：變體「免費但費 2 能量」在不改 core.py 下，換角仍佔
「領航員」類行動格（core.Action.category() 對 subkind='換角' 寫死回 '領航員'、
決策 C5 亦採此讀法）。實測「免費費能量」變體下——

```
換角+領航員攻擊/能力 同回合合法組合數 = 0
換角+移動 同回合合法組合數 = 16
```

——**無法同回合換角＋反擊**，紙上「換角一次切到免疫又不損反擊、故嚴格支配走位」
的核心前提不成立。此為 core.py 既有行為＋決策 C5 的直接結果，任務禁改 core.py。

**退而量測的可實作部分（移動占比）**：能量制換角**是否**壓低移動占比——
見 `moveratio_compare.json`。預期結論：方向一移動占比本就趨近 0（攤牌局大家都龜、
基線 greedy 互打移動占比亦極低），故「免費換角把移動擠到近零」即使觀察到，
也**無法歸因於換角變體**——它在有沒有換角變體下都近零。因此紙上「換角把走位
這層歸零」的因果無法在此模型分離、更遑論「嚴格支配」（前提根本不可實作）。

## new_holes（實測新發現的問題）

1. **【系統性】方向一是安全網攤牌局、攻擊被壓制**：每名 70×3＝210HP＋防禦型
   間穿透率極低 → 連基線互打都拖到中位 41~46 回合、安全網 100% 觸發、擊殺主力
   是安全網而非戰術。這使「猜想樂趣的開局讀心」在多數對局中不承重——勝負由
   攤牌期存活決定。**比任何單一套路更根本**：方向一的核心樂趣三支柱在此數值
   配置下被安全網架空。

2. **【承套路1】回復牆讓安全網反成拖局幫兇**：盾型每回合回 25~55HP >> 安全網
   每回合扣 10，安全網無法縮短回復牆對局；反而因為對手打盾型 0 實傷（無法豁免）
   而幫盾型磨死對手。安全網「防拖局」的設計意圖被回復牆反用（紙上套路1/5 的
   預測命中）。

3. **【承套路5/決策C5】換角變體的紙上前提與引擎落差**：紙上多處假設「免費換角
   不佔行動格」，但 core.py 的 category() 寫死使任何換角變體都佔領航員類——這使
   「免費換角支配走位」這條退化路徑在現行引擎下根本無法成立。設計上若真想測
   這條，必須改 core.py 的行動類別歸屬（本次禁改）。

4. **【承環境】攻擊主動性在此配置是負資產**：三個「主動進攻」套路（2/3/4）全部
   輸給龜縮基線。這反過來揭示一個平衡問題——**方向一目前獎勵純龜縮＋回復、
   懲罰主動輸出**，與「換角預測是猜想樂趣主引擎」的設計目標背道而馳。

## 局數與環境

- **heal_wall**：對 cheap 基線各 1000 局（先後手各半）；positional 500 局；
  oneply/predictor/naive 小 n（best-effort）。
- **shave_heavy / global_focus / focus_snowball / free_swap_counter**：先跑 pilot
  各 20~30 局確立 verdict（勝率 0.00~0.63、遠離 55% 門檻），再補 cheap 基線各
  500 局寫入 `exploit_*_<baseline>.json`（驗證期本機 load average 一度達 **171**、
  8 核心嚴重超額訂閱，來自其他並行 session，1000 局全矩陣壁鐘不可行；500 局
  Wilson 95% CI 半寬約 ±4.4pp，足以把 0.17~0.41 與 0.55 門檻分開）。因 verdict
  由 pilot 已鎖定（效應量極大、補樣本只縮 CI 不翻結論），500 局補樣本於背景
  續跑、其數字落 JSON 供稽核。所有縮減如實記錄、不無聲縮水。
- oneply/predictor/naive 每局 ~85 秒（方向一行動空間巨大、對手 oneply 枚舉昂貴，
  比照 dir1.py 既有效能決策），故僅對主力漏洞 heal_wall 做小 n best-effort。
- 隔離機制數字（先削後重穿透、砲秒盾發數、閃避蒸發率、自攻對力場 0）皆確定性
  或 4000 次抽樣，精確、與紙上完全吻合。
