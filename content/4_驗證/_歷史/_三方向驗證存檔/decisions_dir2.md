## 阻斷級

### 阻斷1：閃避「每點 1/3 機率抵銷一整擊」讀法A vs 讀法B

**發現**：「一整擊」可讀成「一發（一個攻擊次數）」（讀法A）或「一次攻擊宣告（整個單次傷害×攻擊次數）」（讀法B）。實跑差距達一個數量級（閃避8對10×5：讀法A期望實傷23.56、讀法B僅1.98）。

**採用**：讀法A——這已是 `core.py` `resolve_attack` 的既定實作：`negated = sum(1 for _ in range(evasion) if rng.random()<1/3); eff_hits = max(0, hits-negated)`，即每點閃避獨立擲一次，成功數從「攻擊次數」扣、封頂於攻擊次數本身。契約 §一明寫「二項分佈」與此吻合。不採用讀法B（任一點成功即整個攻擊全免），因為：
1. 讀法B 需要修改 `core.py` 共用核心的閃避邏輯，違反「共用檔不得修改」的約束；
2. `tests.py` §6.1 回歸測試已鎖定讀法A 的期望值（`閃避3 機率制 10×5 期望≈40`），且已全數通過；
3. 讀法A 與存活條 1「多發載體有閃避天敵」的精神一致——讀法B 會讓閃避完全免疫多發攻擊，直接違反存活條 1。

**驗證**（實跑，seed=999，N=40000，機率制）：閃避3 對 10×5 期望實傷 39.85（理論值40，`tests.py` 容許誤差<1.0，通過）。方向二游擊型（閃避8）對機關槍5×5／三連彈10×3／電漿彈10×4／亂射10×3 的期望有效傷害另於相剋矩陣（exp3）逐格列出。

### 阻斷2：換位裝置「後排」vs 鉤索「前列」的方向軸不一致

**發現**：換位裝置字面「向其後排推1格」用「排」（橫向），鉤索「向其前列拉1格」用「列」（縱深）；字面若照做，換位裝置會把敵人往橫向推、且「後排」在排2時無法判定（排1或排3都算後）。

**採用**：兩者統一為縱深「列」軸——換位裝置＝命中後把敵人向敵方後排列方向推 1 列（side0 攻擊時敵方朝列6方向、side1 攻擊時敵方朝列1方向）；鉤索＝命中後把敵人向攻擊者側拉 1 列（side0 攻擊時敵方朝列4方向、side1 攻擊時敵方朝列3方向）。純文字修正、不動數值，與規則律師建議一致。`dir2.py` 內以 `_push_row(att_side)` / `_pull_row(att_side)` 兩個輔助函式實作方向計算，取代 `setups.py` 原本 `on_hit=('推',1)` / `('拉',1)` 中未明確定義軸向的部分（`setups.py` 現有實作實際上尚未接上位移邏輯，屬待補實作而非錯誤實作——見下方「setups.py 既有落差」說明）。

**setups.py 既有落差**：檢視 `setups.py` `dir2_chip_db()` 內 #16/#17 的 `ability_face` 含 `'on_hit': ('推',1)` / `('拉',1)` 欄位，但 `Dir2Engine._apply_chip_ability`／`Engine._do_attack` 均未讀取此欄位、未實際執行位移。即目前 `setups.py` 的換位裝置/鉤索**只造成傷害、不位移敵人**。`dir2.py` 補上位移執行邏輯（覆寫 `_apply_chip_ability` 讀取 `on_hit` 欄位並呼叫位移函式），這是補完既有欄位的既定意圖、非新設計。已記入 `results/engine_bugs_dir2.md`。

**驗證**：`dir2.py` 附加測試驗證換位裝置命中後敵方列號朝其半場後排方向 +1（受邊界截斷，見阻斷3）、鉤索命中後敵方列號朝攻擊者側 −1（同樣受邊界截斷）。

### 阻斷3：強制位移越界規則（半場/盤面邊界）缺漏

**發現**：鉤索命中敵方最前列（列4）並拉近1列會進我方半場（列3）；換位裝置推已在列6的敵人會無格可推。規格未定義邊界處置。

**採用**：強制位移同受「不可進入對方半場」（共用基準§二）與盤面邊界（列1–6）雙重約束；位移量被邊界截斷時，目的地截斷到邊界格（即最多推/拉到緊貼邊界的那一列），若起點已在邊界、無格可再移動，則本次位移無效果（單位留在原地），但傷害仍照常結算（位移是「命中後」附加效果，傷害與位移各自獨立判定、位移落空不影響傷害）。此為規則律師建議的直接採用，純規則補全、不新增機制。

**驗證**：`dir2.py` 附加測試驗證：(a) 鉤索命中列4敵人時，目的地截斷在列4（不可進我方半場列3），敵人位置不變但傷害仍結算；(b) 換位裝置命中已在列6的敵人時，位移無效果、傷害仍結算。

### 阻斷4：陷阱「進入」觸發條件與強制位移交互

**發現**：陷阱觸發條件「進入」未定義是否涵蓋「被強制位移到該格」，也未定義「經過但未停留」是否觸發。

**採用**：任何原因（自主移動或被強制位移）使敵方單位「停在」陷阱格即觸發；本模組的移動模型是逐格離散（無「經過中間格」的連續軌跡概念——基本移動與晶片移動面都是直接指定目的地格，共用核心 `core.chip_move_dsts` 未模擬路徑上的中間格），故「經過不觸發」在本模型下自動成立（沒有「經過」這個中間狀態可觸發）。這與規則律師建議一致，且不需要額外程式碼區分「經過」與「停留」——因為引擎的位移模型本身就只有離散的起點與終點，沒有第三種狀態。

**驗證**：`dir2.py` 附加測試驗證：換位裝置/鉤索把敵人強制位移到陷阱格後，`hook_trap_check` 確實觸發（沿用 `setups.py` 既有 `hook_trap_check` 邏輯，該邏輯已對「移動」與「晶片移動」呼叫，本次新增對「被強制位移的終點」也呼叫此 hook）。

### 阻斷5：能量球產生時序全缺（起算回合、計數基準、重選、上限交互）

**發現**：規格「每2回合...產生1顆」未定義從第幾回合起算、計的是全局回合數或各方回合數、選中格已被佔用時如何重選、上限5是否讓產球停止。

**採用**：
1. **起算與計數基準**：全局回合數（`st.round_no`），從第2個全局回合**結束時**起、每2個全局回合各生成一次（`round_no % 2 == 0` 時，於該回合先手方開始行動時觸發——這是 `setups.py` `Dir2Engine.hook_start_turn` 的既定實作：`if side == st.first_mover and st.round_no % 2 == 0`，本模組沿用不改）。
2. **候選格**：雙方半場各自的空格集合（無單位佔據、無既有資源球），若選中偽隨機格已符合此條件即可（`setups.py` 既有實作已用 `free = [(r,c) for r in own_rows(side) for c in COLS if (r,c) not in st.resource_cells and st.units[side].pos != (r,c)]` 排除；本模組沿用）。
3. **選不到空格**：候選集合為空（`free` 為空列表）時該次不產（`setups.py` 既有 `if free: ...` 已隱含此語意，沿用）。
4. **場上球數上限**：規格「資源上限5」讀作**單方持有量上限**（`st.resources[side] < 5` 才能撿——`setups.py` 既有實作已如此，見 `hook_resource_pickup`），非「場上球數上限」；場上未拾取的球數不受此上限限制，生成邏輯每2回合固定各生1顆（不因場上已有多顆球而停止生成，只受「無空格可選」的自然限制）。

**驗證**：`dir2.py` 附加測試驗證：(a) round_no=2 時（先手方開始行動）雙方半場各生成1顆資源；(b) round_no=1,3 時不生成（僅偶數全局回合）；(c) 場上已有5顆球、雙方持有量都是0時，仍會嘗試生成（若有空格）；(d) 單方持有量達5時無法再撿（`hook_resource_pickup` 既有邏輯）。

## 重要級

### 重要1：傷害帶地板10 = 力場帶天花板15 相撞（重甲型近乎免疫低傷害攻擊）

**發現**：力場15（重甲型）讓單次傷害10、15兩檔攻擊完全歸零（0實傷）；力場10（壁壘型）讓單次傷害10歸零。這是存活條1要防的「某防禦沒有天敵」在力場版的具體化——地板10 < 天花板15 使傷害帶存在被完全吃光的攻擊帶值。

**實測**（seed=0，機率制）：field=10 對 dmg=10 → 實傷0；field=15 對 dmg=10 → 實傷0；field=15 對 dmg=15 → 實傷0。方向二 15+3 張晶片中，單次傷害≤15的攻擊共8種（短劍15、機關槍5、三連彈10、追蹤彈15、換位裝置15、鉤索10、電漿彈10、亂射10）加上領航員自身攻擊10×1，對重甲型（力場15）而言全數逐擊歸零（機關槍例外——其5點傷害本身就低於力場5，屬既有共用晶片表缺陷，見下方 engine_bugs_dir2.md）。

**採用**：在 `dir2.py` 的方向二 setup 內，把「重甲型」原型的力場值從規格文字的15**下修為5**（共用基準力場帶{5,10,15}的最低檔），使力場天花板（5）嚴格低於傷害帶地板（10）。壁壘型的力場10維持不變（雖然力場10仍與傷害地板10同值、對單次傷害恰10的攻擊仍是0實傷，但這是「壁壘型」設計本身的攻防取捨、且壁壘型另有護盾25作第二層防禦，非本次決策範圍；重甲型因閃避僅4、無護盾，力場是其唯一有效防禦層，天花板=地板的影響對它更致命，故只調整重甲型）。此為規則律師建議「力場帶天花板 < 傷害帶地板」的最簡實作，僅動方向二 overlay 內的原型數值，不動 `chips.py` 共用傷害帶或 `core.py` 共用力場公式。

**驗證**：實跑（trials=20000，seed=12345）力場5 對方向二全部15種攻擊形狀（含機關槍5×5），僅機關槍單擊5×5 因5≤5仍全額吃光（期望實傷0.00，與地板10無關、是機關槍自身傷害5<10地板的既有問題），其餘14種攻擊形狀對力場5均有嚴格>0的期望有效傷害。相剋矩陣（exp3_相剋矩陣_方向二.json）逐格列出實測值。

**已記錄**：規格文字「重甲型：力場15＋閃避4」與實作值「力場5＋閃避4」的差異已記入 `results/engine_bugs_dir2.md`，供之後與 user 核對規格文字是否需同步更新（本次決策屬「規則律師建議、可逆的數值選擇」，非規格權威本身的修改）。

### 重要2：費2 動態次數+1 對存活條1 多發載體計數的動態失真

**發現**：費2（下一張晶片能力面攻擊次數+1）能把單發變雙發、雙發變三發，繞過閃避剋制的靜態計數（存活條1「次數≥3有4種8張」是照固定卡面算）。

**採用**：在相剋矩陣（exp3）內，除固定卡面攻擊形狀外，另外加一組「費2加成後」的攻擊形狀（把方向二全部15種攻擊形狀的攻擊次數各+1）一併跑相剋矩陣，並在摘要中報告「含費2動態後，仍有多少種形狀對閃避各檔維持非零穿透」。不改變資源系統本身的機制（費2 依然是規格所定義的「下一張晶片能力面攻擊次數+1」），只補驗證維度，不代為修改機制設計。

**驗證**：見 `exp3_相剋矩陣_方向二_費2加成.json`；本檔摘要一併列出。

### 重要3：資源使用（費2/費3）是否佔行動未定義

**發現**：「拾取不佔行動」已明，但「使用資源」是否佔行動槽未定義，直接影響行動經濟的核心假說（移動面被選率）。

**採用**：資源使用為**即時效果、不佔行動**（與拾取一致）——理由：(a) 規格原文「反向利用警戒觀測」段落把資源定位為「位置誘因」而非「另一種行動資源」，若使用要佔行動槽，會讓資源系統與晶片/領航員類直接搶槽、稀釋移動面誘因的設計意圖（規格核心假說「幾何＋場地資源就能把移動撐成真選擇」隱含資源是移動的加成、非額外的行動類別）；(b) 費2「下一張晶片能力面攻擊次數+1」的自然實作是「宣告在下一次使用晶片能力面時生效的修飾符」，與晶片本身的行動槽合併，不需要獨立佔用一個行動；(c) 費3「立即抽2張」若佔行動，等同把「抽牌」包裝成一種新行動類別，與共用基準§六「每回合開始抽1」的被動抽牌機制邏輯層次不同，容易在行動類別判定上產生混淆。

**費3 超手牌上限處置**：費3 立即抽2張時，若已達手牌上限6，抽到上限即止（沿用共用核心 `Engine.draw` 既有邏輯：`if len(st.hands[side]) >= limit: break`），不額外棄牌、不允許超過上限暫存後再棄。

**驗證**：`dir2.py` 附加測試驗證：(a) 使用費2/費3 後，`actions_this_turn` 長度不增加（即該次資源使用不算入本回合已用的行動格）；(b) 手牌6張時使用費3，抽牌後手牌仍為6（不超過上限、不報錯）。

**實作訂正（開發過程自查）**：費2 加成邏輯首版誤放在共用 `_do_attack`（同時攔截領航員自身攻擊與晶片能力面攻擊兩條路徑），與規格原文「下一張**晶片**能力面攻擊次數+1」不符——會讓費2 意外也加成到自身攻擊（10×1→10×2）。開跑實驗矩陣前自查發現並改為只在 `_apply_chip_ability`（晶片能力面攻擊的唯一入口）套用加成，領航員自身攻擊路徑不受影響。`dir2_tests.py` 補上直接對照測試（同一待生效旗標下，先跑自身攻擊確認不變、再跑晶片攻擊確認加成生效且旗標清除）。此訂正在啟動正式 2000/500 局實驗矩陣之前完成，實驗結果不受影響。

### 重要4：legal_actions 是否誤產生換角動作

**發現**：共用 `Action` 型別含「換角」子類，方向二1對1無第二角色，若未顯式關閉，`legal_actions` 可能吐出換角動作。

**採用**：確認 `core.py` 共用 `Engine.hook_change_navi_actions` 預設回傳 `[]`（空清單）。`setups.py` 的 `Dir2Engine` 未覆寫此 hook，因此方向二場上 `legal_actions` 對「換角」子類永遠不會產生任何 Action——這與 `dir3.py`（見 `decisions_dir3.md` 重要4）相同的既有安全網。`dir2.py` 明確不覆寫此 hook（維持繼承共用預設的空清單），並在附加測試中斷言驗證，落實規格律師建議的「方向二 legal_actions 不產生換角動作」。

**驗證**：`dir2.py` 附加測試 `test_dir2_no_navi_change()`：對多個隨機盤面狀態呼叫 `engine.legal_actions(st, side)`，逐一斷言不含 `subkind=='換角'` 的 Action。

### 補充：資源使用需要決策掛點才會真的發生（開發過程自查補完）

**發現**：`use_resource_cost2`/`use_resource_cost3` 首版只是 `Dir2OverlayEngine` 的方法，但沒有任何策略會呼叫——`legal_actions` 不產生「使用資源」的 Action（因為決策重要3 已定案資源使用不佔行動、不走 Action/legal_actions 這條路徑），而共用 `Engine.take_turn` 也沒有為方向專屬的「不佔行動宣告」留通用掛點，導致資源使用在最初實作裡永遠不會被觸發（整個實驗矩陣會全數顯示費2/費3使用總數＝0，resource_aware 策略評分時雖考慮資源價值、卻從未真正花用）。

**採用**：在 `dir2.py` 的 `Dir2OverlayEngine` 覆寫共用 `take_turn`（僅方向二 overlay 內覆寫、不改 `core.py` 本體），在策略選兩行動的前後各留一個資源使用決策點（`policy.decide_resource_use(st, side, rng, engine, timing)`，`timing∈{'before','after'}`）；只有定義此方法的策略（`ResourceAware`）會被呼叫，其餘共用策略（含 `resource_blind`）未定義此方法、維持「視資源為雜訊」的預設行為，不受影響。`ResourceAware.decide_resource_use` 策略：資源≥3 優先費3（回合開始就抽，讓抽到的牌能在本回合用上）；資源界於2–3之間且手上有晶片能力面攻擊可用時才費2（避免加成旗標掛著沒攻擊可用而浪費）。

**驗證**：實跑（seed=42，resource_aware vs resource_blind，游擊 vs 壁壘，first_turn_comp=一行動）單局費2使用8次、費3使用1次，確認資源確實被消耗；`dir2_tests.py` 全部32項測試（含此改動前已通過的位移/陷阱/資源生成測試）與共用 `tests.py` 58項測試改動後重跑仍全數通過，確認 `take_turn` 覆寫未破壞既有行為。此修正在啟動正式規模（2000/500局）實驗矩陣之前完成。
